# Project Design Docs Skill

生成一套完整的项目设计文档，覆盖从系统架构到前后端开发的全部环节。

## 目标文档清单

| 文件名                  | 说明           |
| -------------------- | ------------ |
| `系统设计.md`   | 系统整体架构设计     |
| `接口设计.md`      | API 接口设计规范   |
| `接口联调指南.md` | 前后端 API 联调指南 |
| `后端开发指南.md`   | 后端开发规范与指南    |
| `后端任务清单.md`    | 后端开发任务清单     |
| `前端开发指南.md`  | 前端开发规范与指南    |
| `前端任务清单.md`   | 前端开发任务清单     |

---

## 使用前：收集项目信息

在生成文档前，先向用户确认以下关键信息（未提供的可根据上下文推断，推断内容在文档中用 `TODO` 标注）：

1. **项目名称与简介**：这个系统是做什么的？
2. **技术栈**：前端框架、后端语言/框架、数据库、部署方式
3. **核心模块**：系统有哪些主要功能模块？
4. **用户角色**：有哪些类型的用户/权限？
5. **已有约定**：是否有已定的接口风格（REST/GraphQL）、命名规范等？

如果用户说"直接生成"或"根据上下文来"，则基于已知信息生成，所有不确定项用 `<!-- TODO: 补充 -->` 标注。

## 技术选型
| 层级 | 技术 | 选型理由 |
|------|------|----------|
| 前端 | ... | ... |
| 后端 | ... | ... |
| 数据库 | ... | ... |

## 核心模块划分
[列出主要模块，每个模块一段简短说明]

### 2. `接口设计.md` — API 接口设计

```markdown
# API 接口设计文档

## 设计规范
- 风格：RESTful / GraphQL（按项目选择）
- 基础路径：`/api/v1`
- 认证方式：JWT Bearer Token
- 响应格式：统一 JSON 结构

## 统一响应结构
\`\`\`json
{
  "code": 200,
  "message": "success",
  "data": {}
}
\`\`\`

## 错误码定义
| 错误码 | 说明 |
|--------|------|
| 400 | 请求参数错误 |
| 401 | 未认证 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 500 | 服务器内部错误 |

## 接口列表

### [模块名]
#### GET /api/v1/[resource]
- 描述：
- 请求参数：
- 响应示例：

#### POST /api/v1/[resource]
- 描述：
- 请求体：
- 响应示例：
```

> 根据项目的模块数量，为每个核心模块生成对应的接口组。

### 3. `接口联调指南.md` — API 联调指南

```markdown
# API 联调指南

## 本地开发环境配置
- 后端服务地址：`http://localhost:8080`
- 前端开发代理配置示例

## 认证流程
1. 调用登录接口获取 Token
2. 后续请求在 Header 中携带：`Authorization: Bearer <token>`

## 联调步骤
1. 启动后端服务
2. 配置前端代理
3. 按模块逐一联调

## 常见问题
- CORS 跨域处理
- Token 过期处理
- 文件上传注意事项

## 接口 Mock 方案
[开发阶段如何 Mock 接口数据]
```

## 输出格式要求

- 所有文档输出为标准 Markdown 格式
- 不确定的内容用 `<!-- TODO: 说明 -->` 标注，方便开发者后续补充
- 代码块注明语言类型
- 接口路径、字段名等技术内容使用 `行内代码` 格式
- 如果用户要求生成为文件，使用 `file_create` 工具将每个文档保存为独立 .md 文件
